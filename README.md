# AuthFlows

AuthFlows is a Next.js App Router authentication demo that combines email/password signup, email verification, OTP-assisted login, password reset, protected routes, and a profile dashboard backed by PostgreSQL and Prisma.

The app is built to show a complete account lifecycle, from registration through verification and sign-in to profile editing inside a protected dashboard.

## Core Features

### 1. Account Registration

Users can create an account through the sign-up flow. During sign-up:

- the form is validated on the server
- duplicate email addresses are rejected
- the password is hashed with `argon2`
- a new user row is created in PostgreSQL through Prisma
- an email verification token is generated and stored
- a verification email is sent through Resend

This means a user account is created first, but it is not considered active until email verification succeeds.

### 2. Email Verification

After registration, the user receives a verification link. When they open it:

- the token is validated
- the token expiry is checked
- the user’s `emailVerified` field is updated
- a welcome email is sent on first successful verification
- the user is automatically signed in
- the user is redirected directly to the dashboard

This is an important UX improvement over the older “verify, then manually log in” flow. The current implementation turns verification into the completion step of onboarding.

### 3. Password Login With OTP Verification

This project does not stop at basic email/password sign-in. Instead, it adds a second layer of login verification:

- the user enters email and password
- if the password is correct and the email is verified, a one-time code is generated
- the code is stored in the database with an expiry time
- the code is emailed to the user
- the user is redirected to the `/verify-login` page
- the user must enter the correct 6-digit code to finish the login

This creates a lightweight two-step login flow that improves security without introducing an external authenticator app.

### 4. Password Reset

If a user forgets their password, they can request a reset link:

- the app generates a password reset token
- the token is emailed to the user
- the user opens the reset-password page with the token
- the app verifies the token and expiry
- the new password is validated and hashed
- the user’s password hash is updated in the database

### 5. Protected Dashboard

The dashboard is a protected area available only to authenticated users. It currently shows:

- a personalized welcome headline
- the user’s name
- the user’s email address
- login verification status
- the current profile-picture state

The dashboard is intentionally focused on user-relevant information rather than raw internal identifiers.

### 6. Editable User Profile

Inside the dashboard, users can update a limited set of profile fields:

- full name
- age
- gender
- profile picture

The system is intentionally restrictive. Users cannot edit arbitrary account data. Only explicitly permitted fields are accepted by the server action that updates the profile.

### 7. Profile Picture Upload

Users can change their profile picture from the dashboard using:

- drag and drop
- upload from computer

The uploaded image is resized and compressed in the browser before being submitted. This keeps the payload under Next.js Server Action body-size limits and avoids unnecessarily large requests.

If the user does not provide a custom image, the dashboard falls back to a default avatar generated by the app.

### 8. Welcome Email

When a newly registered user successfully verifies their account for the first time, the system sends a welcome email. This is important because it ties the message to actual account activation, not just account creation.

That makes the email more meaningful and prevents welcome emails from being sent to accounts that were never verified.

## Authentication And Access Flow

The current end-to-end flow looks like this:

### New User Flow

1. User signs up.
2. App creates the account with `emailVerified = null`.
3. App sends an email verification link.
4. User opens the link.
5. App verifies the token.
6. App marks the email as verified.
7. App sends the welcome email.
8. App creates a session.
9. User is redirected to `/dashboard`.

### Returning User Login Flow

1. User enters email and password.
2. App validates credentials.
3. App generates a login OTP and emails it.
4. User is redirected to `/verify-login`.
5. User enters the 6-digit code.
6. App validates the OTP.
7. App creates a session.
8. User is redirected to `/dashboard`.

### Password Reset Flow

1. User requests password reset.
2. App generates a reset token.
3. App emails the reset link.
4. User opens the link.
5. App validates the token.
6. User submits a new password.
7. App hashes and stores the new password.

## How The System Works Internally

### Session Management

Authentication is managed with NextAuth/Auth.js using a JWT session strategy. The app exports:

- `auth`
- `signIn`
- `signOut`
- `handlers`

from [`auth.ts`](./auth.ts).

The project uses a custom `Credentials` provider with multiple sign-in paths:

- standard email/password login
- OTP-completed login using `loginToken`
- post-verification auto-login using `emailVerificationToken`

This gives the app one centralized authentication layer while still supporting multiple entry points into the session system.

### Route Protection

The root route `/` redirects users based on session state:

- authenticated users go to `/dashboard`
- unauthenticated users go to `/login`

The app also uses [`middleware.ts`](./middleware.ts) to guard protected routes more broadly. Public routes such as login, signup, verification, and password reset remain accessible without a session, while protected routes require a valid auth token.

### Database Access

Prisma is used as the ORM layer. The Prisma Client is initialized in [`lib/prisma.ts`](./lib/prisma.ts) using:

- `@prisma/client`
- `@prisma/adapter-pg`
- `pg`

The project uses PostgreSQL as its database backend.

### Email Delivery

Transactional emails are sent through Resend in [`lib/email.ts`](./lib/email.ts). The project currently sends:

- email verification emails
- login OTP emails
- password reset emails
- welcome emails

### Validation

The project uses `zod` to validate form input. Validation logic currently lives in:

- [`lib/validations/signup.ts`](./lib/validations/signup.ts)
- [`lib/validations/login.ts`](./lib/validations/login.ts)

This keeps validation rules close to the authentication domain and avoids fragile client-only validation.

## Auth Flow

### 1. Registration

When a user signs up:

1. The form is validated with `signupSchema`.
2. The app rejects duplicate emails.
3. The password is hashed with `argon2`.
4. A `User` record is created with `emailVerified = null`.
5. A 24-hour email verification token is stored.
6. A verification email is sent through Resend.

### 2. Email verification

When the user opens the verification link:

1. `/verify-email` forwards to `/api/verify-email`.
2. The token is checked for existence and expiry.
3. The user is marked as verified if this is the first successful verification.
4. A welcome email is sent once on first verification.
5. The user is automatically signed in through the credentials provider.
6. The user is redirected to `/dashboard`.

### 3. Login with OTP

The login flow is intentionally two-step:

1. The user submits email and password on `/login`.
2. The password is verified against the stored `passwordHash`.
3. If the account is verified, the app creates a 6-digit code and a `loginToken`.
4. The code is emailed to the user.
5. The user is redirected to `/verify-login`.
6. The code must match the latest unexpired verification record.
7. On success, the user is signed in and redirected to `/dashboard`.

The verify-login screen also supports resending the OTP, with a 60-second cooldown tracked in the database.

### 4. Password reset

If a user forgets their password:

1. They request a reset link on `/forgot-password`.
2. The app creates a `PasswordResetToken` with a 15-minute expiry.
3. The reset link is emailed through Resend.
4. The user opens `/reset-password?token=...`.
5. The token is validated before allowing a reset.
6. The new password is hashed and stored.
7. The reset token is deleted after use.

The reset flow also prevents the user from setting the same password they are already using.

## Dashboard

After authentication, users land on `/dashboard`, which currently includes:

- Account overview cards
- Display name and email
- Member-since date
- Short bio preview
- Editable profile form
- Logout button

### Editable profile fields

The dashboard profile form currently supports:

- `name`
- `age`
- `gender`
- `image`
- `homeAddress`
- `workAddress`
- `bio`

Profile updates are handled by a server action and revalidated with `revalidatePath("/dashboard")`.

### Profile image handling

Profile pictures are selected client-side and compressed before submit:

- Accepted types: PNG, JPG, JPEG, WEBP, GIF
- Max upload size before processing: 5 MB
- Images are resized in the browser
- Compressed images are stored as data URLs in the database

If the user has no image, the app falls back to a generated SVG avatar.

## Environment Variables

Create a `.env` file with these values:

```env
DATABASE_URL=
AUTH_SECRET=
RESEND_API_KEY=
EMAIL_FROM=
APP_URL=http://localhost:3000
```

Notes:

- `DATABASE_URL` should point to your PostgreSQL database.
- `AUTH_SECRET` is used by Auth.js.
- `RESEND_API_KEY` and `EMAIL_FROM` are required for email delivery.
- `APP_URL` should match the base URL where the app is running.

## Architecture Notes

- Auth is centralized in [`auth.ts`](./auth.ts) using a custom credentials provider.
- Sessions use the JWT strategy.
- Middleware protects non-public routes by checking for a valid Auth.js token.
- Email verification, login OTPs, and password reset all use separate database-backed token tables.
- Most mutations are implemented as Next.js Server Actions.

## Current Limitations

- Phone and `phoneVerified` exist in the schema but are not yet exposed in the UI.
- Profile images are stored as data URLs rather than external object storage.
- There is no automated test suite configured yet.
